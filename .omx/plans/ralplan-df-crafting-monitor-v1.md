# RALPLAN：三角洲特勤处物品制造监控 APP V1

## 1. 规划状态

- 模式：`$ralplan` / `$plan --consensus`
- 风险等级：deliberate
- 原因：涉及公众发布、QQ/微信授权、账号数据、后台常驻、通知权限、电量和隐私安全。
- 输入规格：`.omx/specs/df-crafting-monitor-detailed-spec.md`
- PRD：`.omx/plans/prd-df-crafting-monitor-v1.md`
- 测试规格：`.omx/plans/test-spec-df-crafting-monitor-v1.md`
- 当前状态：Architect 审查已完成并要求改进；本文件已合并改进，等待 Critic 审查。

## 2. RALPLAN-DR 摘要

### 2.1 原则

1. 本地优先：敏感登录态、制造数据同步和完成时间计算默认留在用户设备上。
2. 只读边界：APP 只能读取制造状态，不写入游戏数据，不自动化游戏行为。
3. 失败透明：接口异常、登录失效和数据过期必须对用户可见。
4. 低耗电默认：默认策略不追求秒级实时，稳定监控必须由用户显式开启。
5. 可验证交付：每个里程碑都必须有测试证据。

### 2.2 决策驱动

1. 账号安全和公众发布信任：服务器不得成为用户登录态集中风险点。
2. Android 后台现实约束：通知、Widget 和后台调度必须按系统限制设计。
3. API 可行性不确定：小程序接口路径需要先验证，再扩大实现范围。

### 2.3 可行方案

#### 方案 A：原生 Android、本地优先、轻后端可选

做法：

- Kotlin + Jetpack Compose 构建 APP。
- WorkManager 负责保守同步和退避。
- Foreground Service 只在稳定监控模式开启。
- App Widget 展示桌面卡片。
- 本地安全存储保存最小登录态。
- 轻后端只做公告、配置和脱敏诊断。

优点：

- 最符合安全和隐私边界。
- Android 通知、后台和 Widget 能力控制最强。
- 不产生服务器代管登录态的集中风险。
- 便于对国产系统后台限制做设备级适配。

缺点：

- 后台实时性受 Android 系统限制。
- 授权和接口适配需要更多 Android 原生工作。
- 公众设备差异会增加测试成本。

#### 方案 B：跨平台框架 + 原生插件

做法：

- 使用 Flutter 或 React Native 构建 UI。
- 用原生插件封装授权、通知、后台任务和 Widget。
- 业务逻辑部分跨平台共享。

优点：

- UI 开发速度可能更快。
- 如果未来做 iOS，部分业务代码可复用。

缺点：

- V1 明确不做 iOS，跨平台收益不足。
- 通知、前台服务、Widget 和后台限制仍需大量原生插件。
- 插件边界会增加调试和电量问题定位成本。

#### 方案 C：服务器代轮询 + Push

做法：

- 服务端保存登录态并代用户轮询。
- 完成后通过 FCM 或厂商 Push 推送通知。

优点：

- 后台通知及时性更容易控制。
- 手机端耗电更低。

缺点：

- 违反 V1 「服务器不代管登录态、不代用户轮询」边界。
- 显著增加账号安全、隐私、合规和运维风险。
- 依赖 Push 通道和厂商适配。

结论：

- 推荐方案 A。
- 方案 B 保留为未来 UI/跨平台重估选项。
- 方案 C 在 V1 明确无效，除非未来获得官方授权并重做隐私与安全设计。

### 2.4 关键前置门禁

M0 是真正的 Go/No-Go 门禁。任何产品 UI、完整后台系统、通知系统和桌面卡片实现，都不得在 M0 通过前进入正式实现。

M0 必须回答：

- 是否存在用户授权路径；
- Android APP 是否能在用户主动授权后获取足够登录态；
- 官方小程序接口路径是否能只读返回四项制造状态；
- 登录态生命周期、刷新条件和失效信号是否可被识别；
- 是否存在授权或校验步骤无法完成导致流程停止的情况；
- 样例响应是否能脱敏保存并形成 parser fixture；
- 失败和异常是否能归类到可退避、可重授权、等待适配或停止路线。

M0 未通过时，项目不得进入 M1-M5。允许保留规划产物，但执行路线必须暂停或回到需求重定义。

## 3. 架构决策

### 3.1 推荐架构

采用原生 Android 本地优先架构。

核心模块：

| 模块 | 职责 |
| --- | --- |
| `app` | 应用入口、导航、依赖装配 |
| `auth` | QQ/微信授权、登录态状态机、安全存储 |
| `crafting` | 制造设施、制造状态、完成时间领域模型 |
| `datasource` | 小程序接口适配、响应解析、错误分类 |
| `sync` | WorkManager、退避、同步调度、完成窗口关注 |
| `notification` | 通知渠道、常驻通知、完成通知、异常通知 |
| `widget` | 桌面卡片数据绑定和刷新 |
| `storage` | Room/DataStore/安全存储封装 |
| `settings` | 权限、监控模式、退出登录、清除数据 |
| `diagnostics` | 脱敏日志、崩溃诊断、问题反馈 |

### 3.2 本地状态流

1. `auth` 产出本地授权状态。
2. `sync` 根据授权状态和用户设置触发同步。
3. `datasource` 拉取并解析制造状态。
4. `crafting` 归一化四项设施状态。
5. `storage` 保存最新快照。
6. `notification` 和 `widget` 订阅本地快照。
7. 异常状态统一进入退避、重授权或等待适配流程。

### 3.2.1 API 适配层契约

小程序接口必须通过反腐层接入，不允许 UI、通知、Widget 或同步调度直接依赖原始响应。

推荐接口：

```kotlin
interface CraftingRemoteDataSource {
    suspend fun fetchCraftingSnapshot(auth: LocalAuthSession): CraftingFetchResult
}
```

适配层必须包含：

- 版本化响应 DTO；
- 原始响应到领域模型的 parser；
- 脱敏 fixture；
- read-only endpoint allowlist；
- 错误分类：网络错误、授权失效、解析失败、接口变更、服务端异常、未知错误；
- kill switch：当接口结构变化或风险升高时进入「等待适配」状态；
- 禁止写请求和非制造数据请求。

### 3.2.2 凭据保管矩阵

| 凭据/状态 | 来源 | 本地存储 | 过期/刷新 | 是否可离开设备 | 删除语义 |
| --- | --- | --- | --- | --- | --- |
| OAuth code | QQ/微信授权回调 | 默认不持久化 | 一次性使用 | 否 | 使用后立即丢弃 |
| Access token | 官方授权结果 | Android 安全存储 | 按授权结果或实测生命周期处理 | 否 | 退出登录清除 |
| Cookie / Session | 小程序接口链路 | 仅在 M0 证明必要且安全时保存 | 按响应和失效信号处理 | 否 | 退出登录清除 |
| Mini-program session | 小程序接口链路 | 仅本地安全存储 | 失效后重授权或等待适配 | 否 | 退出登录清除 |
| Game token | 如接口返回 | 默认禁止持久化，除非 M0 ADR 证明必要 | 不主动刷新绕过 | 否 | 退出登录清除 |
| 用户展示信息 | 授权返回或接口返回 | 普通本地缓存 | 可随退出登录清除 | 可上传前需另行授权 | 退出登录清除 |

所有凭据均禁止写入日志、崩溃诊断、遥测或后端请求。

### 3.2.3 通知与 Widget 事件模型

通知和 Widget 必须从同一份本地快照和事件表渲染，避免状态漂移。

本地表建议：

| 表 | 职责 |
| --- | --- |
| `crafting_snapshot` | 保存四项设施最新归一化状态、同步时间和数据新鲜度 |
| `completion_event` | 保存完成事件、dedupe key、触发状态和通知发送时间 |
| `sync_attempt` | 保存同步尝试、结果、错误分类和退避状态 |
| `monitoring_state` | 保存用户是否开启监控、监控模式和恢复意图 |

规则：

- 完成通知只由 `completion_event` 触发；
- 常驻通知和 Widget 只渲染 `crafting_snapshot`；
- 设备重启后根据 `monitoring_state` 恢复调度；
- 通知权限关闭时，Widget 和首页必须展示通知不可用状态；
- 数据过期时，通知和 Widget 都必须展示最近更新时间。

### 3.3 后台策略

- 默认低耗电模式：WorkManager + 本地完成时间预测 + 手动刷新。
- 稳定监控模式：Foreground Service + 常驻通知 + 接近完成窗口的更积极同步。
- 不以精确闹钟作为唯一机制。
- 低电量、省电模式和连续失败时主动降频。
- 稳定监控模式下 0 到 5 分钟是 best-effort 目标，不是保证；系统限制场景下必须显示最近更新时间和状态解释。

### 3.4 安全边界

- 本地保存最小必要登录态。
- 日志和诊断默认脱敏。
- 后端不保存可复用登录态。

### 3.5 轻后端拒收规则

如果 V1 实现轻后端，后端必须以 schema 形式拒收敏感字段，而不能只依赖客户端自觉。

拒收内容：

- Token、Cookie、Session、授权码；
- 完整请求头或完整响应头；
- 原始接口响应；
- 用户完整制造数据；
- QQ/微信账号标识和游戏账号标识；
- 任何可代表用户访问制造数据的凭据。

允许内容：

- APP 版本；
- 匿名错误类型；
- 匿名设备/系统版本范围；
- 脱敏崩溃堆栈；
- 接口适配版本号；
- 用户明确允许的匿名诊断信息。

## 4. 实施里程碑

### M0：接口与授权可行性验证

目标：

- 确认 QQ/微信授权链路在 Android APP 内的可行实现方式。
- 确认官方小程序接口路径能读取四项制造状态。
- 明确登录态生命周期、字段结构和异常类型。

产物：

- 接口样例和字段字典；
- 授权流程说明；
- 错误分类表；
- 风险状态复核。
- M0 Go/No-Go ADR；
- 脱敏响应 fixture；
- 凭据生命周期记录；
- 鉴权路径验证记录。

退出标准：

- 可以用测试账号只读获取四项制造状态；
- 未发现必须绕过或破解才能运行的步骤；
- 登录失效可被检测；
- 字段能映射到 V1 领域模型。
- 所有请求均在 read-only allowlist 内；
- 退出登录能清除本地凭据；
- 样例响应可被 parser fixture 重放；
- 若任一条件不满足，M0 判定为 No-Go。

### M1：Android 项目骨架与领域模型

目标：

- 创建原生 Android 项目；
- 建立模块边界；
- 实现制造状态领域模型、本地缓存、首页基本展示。

退出标准：

- 首页可展示模拟/测试数据下的四项状态；
- Unit 测试覆盖领域状态、时间计算和新鲜度策略；
- 架构不依赖后端登录态。

### M2：授权与真实同步

目标：

- 实现授权入口；
- 接入真实只读数据同步；
- 实现安全存储和退出登录；
- 实现同步退避。

退出标准：

- 授权成功后能同步真实四项状态；
- 登录失效后停止后台同步并提示重授权；
- 日志扫描不泄露敏感数据；
- 不存在写入游戏数据路径。

### M3：通知与后台监控

目标：

- 实现通知渠道；
- 实现常驻监控通知；
- 实现完成通知；
- 实现低耗电和稳定监控模式。

退出标准：

- Android 13 通知权限流程可用；
- Android 14 前台服务类型合规；
- 完成通知去重；
- 受限后台场景有明确降级。

### M4：桌面卡片与用户控制

目标：

- 实现 4 x 2 桌面卡片；
- 实现设置页；
- 实现暂停监控、退出登录、清除数据和隐私说明。

退出标准：

- 桌面卡片展示状态和最近更新时间；
- 手动刷新或打开 APP 后卡片更新；
- 数据过期时有明确提示；
- 用户可以完全停止监控并清理本地数据。

### M5：公众发布准备

目标：

- 完成兼容性测试；
- 完成异常降级；
- 完成发布说明；
- 完成脱敏诊断和已知问题列表。

退出标准：

- PRD 和测试规格中的发布前验收项全部通过；
- 兼容性矩阵覆盖 Android 8、13、14 和主流国产系统；
- 发布包说明非官方、只读、本地优先和已知限制。

## 5. Pre-mortem

### 5.1 场景一：接口可行性失败

失败表现：

- 小程序接口必须依赖无法合法复刻的运行上下文；
- 授权链路不能在独立 Android APP 中完成；
- 校验流程无法稳定通过。

预防：

- 把 M0 作为硬门槛；
- M0 不做产品 UI 大投入；
- 发现校验流程无法完成时记录状态并暂停该路线。

恢复：

- 将产品降级为规格暂停状态；
- 保留本地通知/Widget 架构设计；
- 等待合规可行接口或用户重新定义边界。

### 5.2 场景二：后台通知不稳定导致用户失望

失败表现：

- 国产系统频繁杀后台；
- WorkManager 被延迟；
- 通知完成延迟超过用户预期。

预防：

- 明确低耗电和稳定监控两种模式；
- 常驻前台通知作为稳定模式前置；
- 桌面卡片和通知都展示最近更新时间。

恢复：

- 增加设备级引导；
- 加强「系统限制」提示；
- 调整默认频率和完成窗口关注策略。

### 5.3 场景三：敏感信息泄露

失败表现：

- 日志或崩溃诊断包含 Token、Cookie、Session；
- 退出登录未清除本地敏感状态；
- 轻后端误收账号数据。

预防：

- 安全测试纳入发布前验收；
- 统一日志脱敏层；
- 后端接口白名单和字段拒收策略；
- 退出登录测试必须自动化。

恢复：

- 立即停用相关诊断通道；
- 发布热修；
- 提供清除数据指引；
- 复盘并补充安全测试。

## 8. ADR

### Decision

V1 采用原生 Android、本地优先架构。服务器只作为可选轻后端，不保存登录态，不代用户轮询制造数据。

M0 作为强制 Go/No-Go 门禁。M0 通过前，不进入完整产品实现；M0 失败时暂停或重定义产品。

### Drivers

- 公众发布下的账号安全和隐私信任；
- Android 通知、后台和 Widget 能力需要原生控制；
- 小程序接口可行性存在不确定性，必须先验证；
- 用户已接受常驻前台通知换取更稳定后台监控。

### Alternatives considered

- 跨平台框架 + 原生插件：暂不选择，因为 V1 不做 iOS，且核心能力仍高度依赖 Android 原生。
- 服务器代轮询 + Push：V1 明确拒绝，因为违反服务器不代管登录态和不代用户轮询边界。

### Why chosen

方案 A 在安全边界、实现可控性、用户信任和 Android 系统能力之间最平衡。它牺牲了一部分绝对实时性，但符合 V1 对本地优先、低耗电和只读监控的约束。

### Consequences

- V1 需要接受 Android 后台限制带来的通知延迟。
- 测试必须覆盖多 Android 版本和国产系统。
- API 可行性验证成为第一里程碑的硬门槛。
- 后续如果需要服务器代轮询，必须重新进入需求和隐私安全设计。

### Follow-ups

- 完成 M0 接口可行性验证；
- 写入 M0 Go/No-Go ADR；
- 确定 Android 最低版本；
- 确定是否使用 Jetpack Glance；
- 确定轻后端是否延后；
- 为公众发布准备非官方和只读边界说明。

## 9. 可用 Agent 类型清单

- `architect`：系统设计、边界、接口和长期权衡审查。
- `critic`：计划质量、风险、验收标准和验证路径审查。
- `dependency-expert`：SDK、库和框架选择。
- `researcher`：官方文档和上游资料确认。
- `executor`：实现具体模块。
- `test-engineer`：测试策略、自动化和兼容性测试。
- `security-reviewer`：安全、隐私和敏感数据审查。
- `designer`：首页、通知、Widget 和设置页交互设计。
- `verifier`：完成证据和发布门禁验证。

## 10. 后续执行编组建议

### 默认：`$ultragoal`

适合把 M0 到 M5 转为可追踪的 durable goal，并按阶段验证完成证据。

建议：

- 主线目标：先 M0，再 M1-M5。
- 每个里程碑必须产出验证证据。
- 不允许跳过 M0 直接实现完整产品。

### 并行：`$team` + `$ultragoal`

当 M0 通过后，可用 Team 并行推进：

- executor：Android 骨架、领域模型、本地缓存。
- executor：通知、前台服务、WorkManager。
- executor：Widget 和设置页。
- test-engineer：测试框架和兼容性矩阵。
- security-reviewer：日志脱敏、安全存储和后端边界。
- designer：用户流程、通知文案和 Widget 信息层级。

Team 产出的证据应回填到 Ultragoal 检查点。

### 明确 fallback：`$ralph`

仅当用户想要单一持久执行者逐项推进并反复验证时使用。当前不推荐作为默认执行路径。
